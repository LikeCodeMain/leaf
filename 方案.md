### **「逻辑-渲染分离架构」**

（英文：**Logic-Renderer Decoupling Architecture**）

但在前端工程领域，它更常被具体地称为以下几种命名方式，取决于上下文和强调的重点：

------

### ✅ 命名

#### . **Headless UI 架构**（最贴近你的描述）

- **含义**：提供**无样式、无 DOM 的纯逻辑层**（即 "headless"），由上层决定如何渲染（Vue / React / Web Components / 原生 DOM）。
- **代表项目**：
  - Headless UI（by Tailwind Labs）
  - Radix UI Primitives
  - React Aria（Adobe）

------

### 📋 架构核心特点

#### ✅ 优势

1. **框架无关性**：核心逻辑可复用于 Vue、React、Svelte 等不同框架
2. **样式灵活性**：完全由开发者控制样式，不受组件库设计系统限制
3. **可访问性内置**：逻辑层通常包含 ARIA 属性、键盘导航等无障碍功能
4. **可组合性强**：基础组件可灵活组合，构建复杂 UI
5. **维护成本低**：逻辑与样式分离，便于独立更新和维护

#### ⚠️ 适用场景

- ✅ 需要高度定制化的设计系统
- ✅ 多框架共享组件逻辑
- ✅ 品牌风格要求严格的项目
- ✅ 需要细粒度控制组件行为

#### ❌ 不适用场景

- ❌ 快速原型开发（需要完整样式）
- ❌ 团队缺乏设计能力（需要现成的样式方案）
- ❌ 对样式一致性要求不高的内部工具

------

### 🏗️ 实现要点

#### 1. **逻辑层设计**

- 提供 **Hooks / Composables** 暴露状态和逻辑
- 处理**状态管理**（打开/关闭、选中状态等）
- 实现**事件处理**（点击、键盘、焦点等）
- 内置**可访问性**（ARIA、键盘导航）

#### 2. **渲染层设计**

- 提供**默认渲染函数**（可选）
- 支持**插槽/Children**自定义渲染
- 暴露**DOM 结构接口**（ref、事件绑定点）
- 允许**样式注入点**（className、style props）

#### 3. **类型安全**

- 完整的 **TypeScript 类型定义**
- 泛型支持不同数据类型
- 严格的 Props 类型约束

------

### 🔄 与其他架构对比

| 特性 | Headless UI | 传统组件库 | 设计系统组件 |
|------|------------|-----------|-------------|
| 样式控制 | ✅ 完全控制 | ❌ 受限 | ⚠️ 部分可定制 |
| 框架支持 | ✅ 多框架 | ⚠️ 单一框架 | ⚠️ 单一框架 |
| 学习曲线 | ⚠️ 较高 | ✅ 较低 | ✅ 较低 |
| 开发速度 | ⚠️ 较慢 | ✅ 较快 | ✅ 较快 |
| 定制能力 | ✅ 极高 | ❌ 低 | ⚠️ 中等 |

------

### 💡 实际应用建议

1. **渐进式采用**：从简单组件（Button、Input）开始，逐步扩展到复杂组件（Select、DatePicker）
2. **建立设计令牌**：定义颜色、间距、字体等设计令牌，确保样式一致性
3. **文档完善**：提供清晰的 API 文档和使用示例
4. **测试覆盖**：重点测试逻辑层的状态转换和事件处理
